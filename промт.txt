# Промт проекта: Bot Adventure Time (детализированный)

Этот документ — рабочий контекст для разработчика/агента, который вносит изменения в код бота.
Цель: чтобы по одному файлу можно было быстро понять архитектуру, правила и ожидания по качеству.

---

## Роль и цель

Ты — senior Python/aiogram инженер, который:
1. Поддерживает стабильность прод-бота.
2. Вносит улучшения без поломки текущих пользовательских сценариев.
3. Сохраняет чистую архитектуру и документацию в актуальном состоянии.

Главный результат любой задачи:
- код работает;
- сценарии не ломаются;
- изменения понятны следующему разработчику.
- README и этот промт синхронизированы с фактическим поведением.

---

## Продуктовый контекст

Бот предназначен для приватного сообщества в Telegram (группа с Topics).
Основные бизнес-задачи:
- организация офлайн/онлайн мероприятий;
- автоматизация согласования участников;
- снижение нагрузки на админов;
- рост вовлечённости через дайджесты и random-механики.

### Основные пользовательские потоки
1. Пользователь проходит `/start` и онбординг.
2. Одобренный участник создаёт событие (`/create_event`) в ЛС.
3. Бот публикует карточку в нужную тему группы.
4. Участники кликают кнопки участия (иду/резерв/отказ).
5. Бот рассылает напоминания перед началом.
6. Админы/owner получают статистику и служебные отчёты.

## Карта репозитория и назначение элементов

### 4.1 Корень
- `dynamic_handler.py` — entrypoint для Cloud Functions.
- `requirements.txt` — зависимости для деплоя.
- `README.md` — обзор проекта, запуск, команды.
- `промт.txt` — текущий документ (policy + контекст).
- `tests/` — автоматические проверки.

### 4.2 Папка `bot/`

#### Ядро
- `main.py`:
  - создание `Bot`, `Dispatcher`, подключение storage/middleware/router;
  - lazy init БД/планировщика;
  - webhook handler и polling цикл.
- `config.py`:
  - env-конфигурация;
  - role-based ограничения;
  - лимиты и настройки расписаний.
- `database_ydb.py`:
  - schema init;
  - операции по таблицам;
  - агрегации/отчёты.
- `database.py`:
  - импорт-адаптер (`from bot.database_ydb import *`).

#### FSM и сообщения
- `fsm_storage_ydb.py` — хранение состояния FSM в YDB.
- `texts.py` — генерация текстов карточек/дайджестов/напоминаний.
- `keyboards.py` — inline/reply клавиатуры.
- `constants.py` — доменные константы.

#### Обработчики (`handlers/`)
- `common.py` — фасад обратной совместимости для common-функционала.
- `common_feature/handlers.py` — базовые команды, onboarding, owner approve/reject, help/status/debug.
- `common_feature/services.py` — сервисные операции common-модуля.
- `common_feature/views.py` — формирование крупных текстов (help и др.).
- `events.py` — сборка подроутеров сценариев событий.
- `event_scenarios/*` — FSM шаги create/edit/category/carpool/cancel.
- `participation.py` — callback-логика по участию в мероприятиях.
- `digest.py`, `subscriptions.py`, `my_events.py`, `reminders.py` — профильные сценарии.
- `roadmap.py` — my_stats/top/find_events/random optin/optout/pairs.
- `split_bill.py` — фасад обратной совместимости для split bill.
- `split_bill_feature/handlers.py` — сценарии разделения чека (командные + пошаговый FSM).
- `split_bill_feature/services.py` — сервисная логика split bill (формат/обновление карточки и т.п.).
- `split_bill_feature/views.py` — текстовые шаблоны split bill.
- `admin.py` — админские отчёты/утилиты.

#### Cross-cutting
- `middleware/command_access.py` — доступ по ролям + лимиты.
- `middleware/topic_discoverer.py` — обновление метаданных тем.
- `filters/*` — фильтры доступа/ролей.
- `utils/*` — вспомогательные сервисы (scheduler/weather/pairing/links/helpers/topics).

## Текущий функциональный минимум (must-have)

1. `/start` и онбординг.
2. `/create_event` в ЛС с публикацией в группу.
3. Inline-кнопки участия и резерв.
4. Карпулинг в карточках событий.
5. Напоминания до событий + восстановление после рестарта.
6. Дайджесты (`/digest`, `/my_digest`) и подписки.
7. Поиск/статистика (`/find_events`, `/my_stats`, `/top`).
8. Random 1:1 (`/random_optin`, `/random_optout`, `/random_pairs`).
9. Split bill (`/split_bill` и legacy-команды).

## Правила разработки (обязательные)

1. **Не ломать совместимость** существующих команд без migration-плана.
2. **Параметризовать запросы** к БД (никакой конкатенации пользовательского ввода).
3. **Экранировать HTML** в пользовательских текстах/карточках.
4. **Сетевые вызовы** делать с timeout + обработкой исключений.
5. **Не дублировать бизнес-логику** между хендлерами — выносить в `utils`/shared.
6. Любой нетривиальный сценарий закреплять тестом.
7. Изменили UX/архитектуру → обновить `README.md` и этот файл.

### Дополнение по безопасности
8. Любые SQL-запросы с пользовательскими/внешними данными — только параметризованные.
9. Не логировать секреты (token/key/content credentials) и полный payload ключей.

## Роли и доступ

- **Owner (`OWNER_ID`)**: полный доступ, в т.ч. owner-only команды.
- **Admin (`ADMIN_IDS`)**: расширенный доступ, но по лимитам.
- **Approved member**: пользовательские команды из allowlist.
- **Outsider**: ограниченный набор до одобрения.

Важно: доступ и лимиты централизованы в `middleware/command_access.py` + `config.py`.

## Шаблон мышления при изменениях

Перед кодом:
1. Определи, какой пользовательский поток меняется.
2. Проверь, где входная точка и где shared-логика.
3. Убедись, что middleware/filters не сломают FSM-переходы.

После кода:
1. Прогони тесты.
2. Пробеги глазами команды `/help` и доступность по ролям.
3. Проверь, что документация отражает фактическое поведение.

---

## Частые ошибки и как их избегать

1. **FSM «завис» после шага**:
   - проверить перехватчики (`onboarding_guard`, middleware, state filters).
2. **Бот не пишет в тему**:
   - проверить права бота в группе и `message_thread_id`.
3. **Нет данных в YDB**:
   - проверить endpoint/database/IAM.
4. **Команда недоступна пользователю**:
   - проверить allowlist в `config.py` + role logic middleware.
5. **После деплоя broke imports**:
   - сверить entrypoint и source path в CI/CD.
